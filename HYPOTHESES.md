# Pre-registered hypotheses — Milestone 1: The Falsification Rig

Written **before** any candidate code (C or assembly) was generated or run.
Verdicts against these predictions go in `results/RESULTS.md` and may not be
edited here after the fact.

## Background claim under test

> LLMs may be able to code effectively in raw machine code, because the costs
> that make assembly laborious for humans (memory, encoding trivia, unreadability)
> do not transfer to LLMs in kind.

The null hypothesis: token economics and training-data distribution mean an LLM
writing C compiled at `-O3` beats the same LLM writing x86-64 assembly directly,
on both correctness and performance.

## Experimental conditions

For each of 10 kernels (spec in `kernels/<name>/spec.md`), the same LLM
(Claude, in-session) produces:

- **A — C baseline:** C from the spec, compiled `clang -O3` (also `-Os` recorded).
  One submission; no feedback.
- **B — Asm one-shot:** x86-64 assembly from the spec, written without executing
  anything. The first run is the recorded result (honesty protocol below).
- **C — Asm iterated:** starting from B's submission, up to **5 feedback rounds**;
  each round the generator sees failing public vectors, register/output diffs,
  disassembly, and instruction counts. Holdout vectors are never shown.

All blobs execute identically inside the Unicorn emulator (never natively).
Metrics: correctness on **holdout** vectors; rounds-to-correct; **dynamic
instruction count** summed over the holdout set (primary perf metric); code
size in bytes (secondary).

## Predictions

- **P1 (C fluency).** Condition A is holdout-correct on ≥ 9/10 kernels.
- **P2 (one-shot asm fragility).** Condition B is holdout-correct on 3–7/10
  kernels, with a lower pass rate on the loopy/stateful kernels (crc32,
  utf8_validate, sort16, rle_encode, base64_encode) than on the straight-line
  kernels (popcount, clamp, bitrev64, isqrt, memchr).
- **P3 (feedback closes the gap).** Condition C reaches holdout-correct on
  ≥ 9/10 kernels within the 5-round budget.
- **P4 (compiler perf dominance, with cracks).** `-O3` beats the final asm on
  dynamic instruction count on ≥ 6/10 kernels — but the asm matches or beats
  `-O3` on ≥ 2 kernels (expected: bit-trick kernels such as popcount/bitrev64).
- **P5 (size).** The final asm blob is smaller than the `-O3` `.text` on ≥ 6/10
  kernels (no vectorization/unrolling bloat).

## Protocol and known limitations

- **Subject = experimenter.** The same model writes candidates and runs the rig.
  Mitigations: predictions fixed here first; holdout vectors generated by
  reference implementations (`kernels/genvectors.py`) and never fed back; the
  one-shot condition is written before anything is executed for that kernel.
- Candidates are written from `spec.md` alone — not from the reference
  implementations, which exist only to produce test vectors.
- A kernel "passes" only if **all** holdout vectors pass.
- This measures one model, one ISA, ten small kernels — evidence about the
  hypothesis's direction, not a general verdict.
